# Chipiona Cipher 
## First SHAX cipher
### Designed and written by Guillermo Amodeo Ojeda.

This is a very simple stream cipher where the actual data to encode is used to generate the key to encode it. 

The cipher is not modelled after a mathematical function like Feistel or Galois, instead I tried to mimic a one-time-pad system, where random data is used as a xor key for the plaintext, being the key as long as the plaintext itself. This kind of cipher is regarded as unbreakable.

Making a real one-time-pad system requires to solve a problem which was kind of solved until the apparition of Quantum Computers and the now infamous -but great- Shor's algorithm: Key distribution. 

So, I limited myself -at the moment- to solve a problem that most ciphers have today: The more data you cipher, the bigger is the exposure of your key, and bigger the risk for it to be broken. 

This may be bad for data-at-rest, but is much worse for data-in-motion, where connections need to last for long time in order to transport a lot of data: remote backup systems, bridges between branch offices that cross internet or remote virtual computing systems, as the ciphertext is easily available for replay attacks.

Therefore, I set my goal as to create a cipher that could encode inmense amounts of data by just having to exchange the key once, and from then on, new keys will be generated by the cipher itself instead of having to be exchanged -or reconnected- and achieve the one-time-pad property that every piece of data is encoded by a different key.

Now, like the key needs to change all the time, and be as long as the plaintext, Why not use the plaintext itself? For sure it fullfills the requirements and you do not need to generate it, as its already available. The only problem is how to make it usable as a key without exposing it, which means to turn it into random data to be used as key.

Instead of reinventing the wheel, I decided to use few well-known techniques and algorithms to achieve my goal, and the result is a *new?* kind of cipher which I call **SHAX**: **S**ubstitute, **H**ash **A**nd **X**or.

As said, that is the kind of cipher, because the name of the first cipher of this kind is Chipiona, because it was in my house in this coastal town in Cadiz (Spain) where I invented it and wrote the first implementation.

## Let me explain how it works:
          
Like I needed random data as key, I decided to use s-boxes filled with pseudo-random integers generated using the key as seed. This is as random as you can go, because as the key changes, so will the integers in the s-boxes. Of course we could not go more random because ciphers must be deterministic enough for the other side to decipher the data.

Also, I decided to use few s-boxes instead one so the same byte of plaintext will use a different s-box value depending on its position on the block. That is, if we use 4 s-boxes (my first idea), the same byte of plaintext will take the same value when in positions 0,4,8,12, but will not be the same if in any other position, of course this is the same for positions 1,5,9,13; 2,6,10,14 and 3,7,11,15.

On the other hand, if we use 16 s-boxes (a much better idea), the same plaintext byte will use a diferent s-box value in each position of the 16 byte block that the cipher uses. But, then I had a better idea: Why instead of choosing the amount of s-boxes from the start (my third idea), is the plaintext itself what determines how many s-boxes are used in each round of block encoding? 

This will further complicate the predictability of the s-boxes used in each round, as it will be 16, 8 or 4 depending on the last byte of plaintext of each round. 

Why use 16 byte blocks in a stream cipher? Well, in order to make sure that the one-time-pad keys are changing all the time and that the same plaintext value do not end up with the same ciphertext value, I hash the s-boxes values with MD5 which outputs a 16 bytes hash that I use as key to encode 16 bytes of plaintext. If the remainder is smaller than 16 the same amount of bytes of the hash are used and the rest of discarded.

I assume that at this point you are guessing why the s-boxes are 32 bits (4 bytes) integers. Yeah, you are right, they are 4 bytes values because MD5 function converts 64 bytes of data to a 16 byte hash, so by mapping each plaintext byte of the 16 bytes block to s-box values of 4 bytes, I obtain the 64 byte needed for a full round of the MD5 function, which will give me a 16 byte hash to use as the xor key for the 16 byte (or less) block of plaintext.

This results in a pretty 'random' key which can actually be built by the other side. But... How does the other side build it if it doens't have the plaintext data until it deciphers the chiphertext?

Well, this is where the original key material plays again. This key is not only used to seed the pseudo-random generators to build the s-boxes, it is also used as the very first input to the cipher in order to generate the first key, which will be the same on both sides after initialization.

After this, the encoding side passes the first plaintext through S-BOX+MD5 function to generate the next key and then XOR's it with the first key to generate the ciphertext. The decoding side then first XOR's the data to obtain the plaintext and then passes it through the S-BOX+MD5 function to create the next key.

Using the original key in this way does not risk it, as the key itself is never used but hashed with the S-BOX+MD5 function to encode the first bit of the plaintext, and then never used again, as the plaintext becomes the next key after S-BOX+MD5 hash.

An important thing to note, is that this use of the data as key for the next data, effectively makes this cipher a CBC cipher as well, but without the risk of pad oracle attacks as there is no need for padding. Not even in the MD5 function. ;-)

For this very reason, the cipher doesn't reset the data pointer position in the MD5 buffer after finalizing the MD5 hash for each block of 16 bytes. This is important because the bit counter is added to the MD5 after the block is processed, which usually results in the MD5 buffer being left with some data, which in turn will be completed with the next data to cipher, in order to generate a new key after applying the S-BOX+MD5 function.

The pseudo-random function used to build the s-boxes is Mother from George Marsaglia, and the code for it is derived from public domain code he published himself when he was alive. 

Mother is a fascinating pseudo-random function, as the numbers that it produces are quite well distributed bitwise, while the pseudo-random function is actually very small.

MD5 was chosen over other hashing routines for the size and speed, and the code for it was written by me following the guidelines and code from RFC-1321 and the Wikipedia page for MD5.

As you could foresee, I know pretty well that MD5 is considered broken because collisions can be generated at will and some pre-image attacks are feasable, but here is used to compress 64 bytes of random data to a 16 byte scrambled key with a good bitwise distribution, and the promise that the original 64 bytes  cannot be recovered easily from the resulting hash. As far as I know, MD5 is still a very good algorithm to achieve that.

Chipiona is small. It takes around 10 Kbytes in code, and around 30 Kbytes of memory after initialized. None of these numbers is big enough to discorage its use in small or embedded systems like routers or bridges.

The best thing is that that you can build your own SHAX cipher by using a different pseudo-random function and hash algorithm e.g, Mersene-Twister and SHA256. 

## To build it:

For Microsoft C/C++ for Windows (Visual Studio C++) use:  

```
nmake /f msc.mak test
```

For Macintosh, Linux and others using GNU C:

```
make -f gcc.mak test
```
### To clean the builds:

Do the same that for build it but use *cleanit* instead of *test* as the target.
